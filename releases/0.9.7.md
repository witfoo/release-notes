# WitFoo Analytics v0.9.7

**Release Date:** 2026-05-29

## Summary

WitFoo Analytics v0.9.7 is a security-hardening and operational-resilience release. It completes a comprehensive, whole-repository security audit, remediates a large batch of upstream dependency CVEs, hardens service startup and container self-healing, and overhauls operator documentation. This release consolidates the hardening work delivered across the 0.9.4–0.9.7 line into a single recommended upgrade.

**Security hardening release — upgrade recommended.**

## What's New

### Security & Hardening

Following a multi-phase security review and a whole-repository audit, v0.9.7 closes a broad set of findings. No exploit details are published here; the categories below describe the protections now in place:

- **Tenant isolation** — All multi-tenant data access now derives the active organization from the verified session, never from client-supplied request parameters. Single sign-on (SAML and LDAP) user lookups are scoped per-organization so an identity in one tenant cannot match an account in another.
- **Conductor trust boundary** — The Conductor management UI is reachable only through the authenticated reverse proxy; direct access is removed and internal trust headers are gated behind an IP allowlist (configurable via `WF_TRUSTED_PROXIES`).
- **Mandatory secret-key enforcement** — `JWT_SECRET` and `AUTH_CONFIG_ENCRYPTION_KEY` are now fail-closed: services refuse to start on an empty or known-default value rather than silently falling back to an insecure default. Stored credentials (LDAP, SAML, TLS keys) and AI-provider API keys are encrypted at rest under a single consolidated XChaCha20-Poly1305 key.
- **WebSocket authorization** — WebSocket upgrade requests are authenticated before the connection is established (token-less upgrades are rejected), and origin checking is tightened across real-time endpoints.
- **Content & attachment safety** — Attachment and user-rendered content paths are hardened against script-injection (XSS) risks.
- **Input & request integrity** — User-supplied values are correctly encoded before any internal request, and previously unbounded query limits are now capped.
- **Crash resilience** — Background workers recover from unexpected panics instead of crash-looping the container, and graceful shutdown no longer surfaces as a spurious fatal error in logs.
- **Log & path hygiene** — Log output is sanitized and on-disk paths (including the redaction alias store) are containment-checked.

### Operational Resilience

- **Startup-race sweep** — Service initialization across the agent and pipeline services was reworked from edge-triggered readiness signaling to a level-triggered model, eliminating a class of startup deadlocks (root cause of intermittent streams-manager and broker-health stalls).
- **Container environment-drift self-healing** — When a WFA upgrade introduces a newly required environment variable (such as an encryption key), any already-running container that is missing it is now automatically recreated within ~60 seconds — closing a crash-loop class that previously required manual intervention.
- **Missing-container self-healing** — A configured service container that goes missing (for example after a transient registry timeout) is automatically recreated, and image pulls fall back to a present local image when the registry is briefly unreachable.

### Conductor & Pipeline

- **Conductor redaction pipeline** — A new opt-in redaction stage tokenizes PII (credit cards, US SSNs, emails, AWS keys, JWTs, PEM private keys, IBANs, plus operator-defined patterns) before artifacts are exported to downstream SIEMs, with deterministic tokens, encrypted at-rest alias storage, and a UI for managing patterns. Requires `REDACTION_MASTER_KEY`.
- **Conductor WebSocket connectivity** — Fixed TLS trust and header-auth handling for Conductor WebSocket/API traffic on AIO+Conductor deployments.
- **Tenable.io REST parser** — Tenable.io asset-inventory data collected via the REST API is now parsed (previously classified as unknown).

### Infrastructure

- **Supply-chain remediation** — Upgraded to **Go 1.26.3** and refreshed the vendored dependency tree (x/net, x/crypto, go-jose, and others), closing 22 known CVEs.
- **WFA bumped to v2.1.17** — See the [WFA v2.1.17 release notes](wfa-2.1.17.md).
- **Build & release discipline** — New automated guardrails prevent the "passes locally, fails in CI" dependency-pin drift class, and three new operator runbooks were added (build discipline, submodule pin management, and environment-variable drift recovery).
- **Documentation overhaul** — New encryption/secret-key configuration reference and architecture-pattern documentation.

## Upgrade Notes

- **Secret keys are now required.** `JWT_SECRET` and `AUTH_CONFIG_ENCRYPTION_KEY` must be set on the API and Incident Engine. WFA appliances generate these automatically; standalone/Compose deployments should use `scripts/dev/generate-secrets.sh` (or `openssl rand`). A node that crash-loops on a missing key immediately after upgrade will self-heal within ~60 seconds on WFA 2.1.13+; on older WFA, run `systemctl restart wfad`.
- **No database migration** is required.
- **No breaking API changes.**
- Conductor redaction is **opt-in** and inactive unless enabled and supplied with `REDACTION_MASTER_KEY`.
